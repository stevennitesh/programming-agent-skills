# Promotion Flow

Use this flow to keep research, synthesis, validation, and runtime behavior
from blurring together.

```text
research finding
  -> synthesis decision
  -> validation plan or evidence
  -> compressed runtime skill wording
```

## Boundaries

- `docs/research/` asks what source pressure and candidate language might matter.
- `docs/synthesis/` decides what the skill pack should become.
- `docs/validation/` records whether changed behavior worked.
- `skills/current/` contains active runtime instructions.

## Promotion Rules

Promote research into synthesis when it becomes a chosen direction, target
spine, wording candidate, or skill design decision.

Promote synthesis into runtime only after the wording is compressed, checked
against existing contracts and skills, and approved for a `SKILL.md` edit.

Promote runtime behavior into validation when there is evidence from a real
task, transcript, fixture, command, or review pass.

If a note only names sources, search terms, or candidate language, keep it in
research. If it says what should change, move it to synthesis. If it says what
Codex must do now, it belongs in a runtime skill.
