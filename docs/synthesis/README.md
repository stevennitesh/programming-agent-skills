# Synthesis

Synthesis docs are the decision-and-evidence bench between source discovery
and final skills. They preserve decision-changing source pressure,
current-runtime evidence, ownership collisions, material alternatives,
selected behavior, extraction placement, and validation design.

Runtime skills should compress hard. Synthesis retains the provenance needed
to reconstruct why behavior won without preserving duplicate presentation or
raw chronology. See
[ADR-0007](../adr/0007-synthesis-preserves-exhaustive-research-runtime-skills-compress.md).

Research owns source meaning and evidence. Synthesis owns downstream design
judgment. Runtime remains executable authority.

The general outside-source loop remains optional:

```text
source question
  -> optional facet map
  -> search and verify
  -> extract concepts and techniques
  -> distill source packet
```

A Deploy Campaign always performs its bounded Research Pass.

| File Or Folder | Role |
| --- | --- |
| [`methods/`](methods/) | Reusable deployment and source-distillation methods |
| [`skill-context-relationships.md`](skill-context-relationships.md) | Current context pointers, cross-skill relationships, and boundary owners |

## Synthesis Ownership

Give every decision one owner:

| Decision | Owning synthesis |
| --- | --- |
| A skill's outcome, admission, authority, process, outputs, completion, and use of another capability | `skills/<consumer-skill>.md` |
| Exact changes to one skill's runtime, references, helpers, templates, tests, evaluations, and installation surfaces | `skills/<file-owner>.md` |
| A provider capability consumed by another skill | Consumer records required outcome; provider records concrete representation and verification |
| One invocation or Return edge | Participating syntheses own their boundaries; the relationship index records the accepted edge |
| Repo-setting schema and reconciliation behavior | `skills/repo-bootstrap.md` |
| One target repository's literal settings | That repository's approved setup state |

Use **consumer requirement, owner implementation**. A consumer names why and
when it needs a capability, the observable result, and its reaction when
absent. It does not prescribe the provider's implementation. Cross-document
references are contracts, not copies.

Synthesis is design authority only until implemented. Canonical runtime files
remain executable authority; target-repository settings remain instance
authority.

## Canonical Extraction

Start manually with Deploy Prompt 1, or use `Run Deploy Campaign on <skill>`
for one fresh campaign epoch:

```text
Prompt 1: freeze intent-derived M0 specification
  -> fuzzy contract: Conditional Behavior Decision Interlude
Research Pass: independent discovery, then upstream/current verification
Prompt 2: reconcile evidence and design H1
  -> contract change: Conditional Behavior Decision Interlude
  -> technical choice: Conditional Prototype Interlude
Prompt 3: materialize exact M0 and H1
Prompt 4: prove M0, test H1, freeze V1
Pruning Pass: derive P1 without V1 regression
Prompt 5: promote and install P1
optional Prompt 6: Git delivery
```

The campaign root retains transitions, verification, and user interaction.
Fresh-context unit owners execute one unit and stop. Exact evidence is reusable
only when bytes, claim, task, model, host, protocol, configuration, tools,
authority, evidence, runtime, rubric, and proof lane match. A prior campaign
cannot skip a unit in a fresh epoch.

### Runtime identities

| Identity | Meaning |
| --- | --- |
| M0 | Intent-derived, behavior-minimal runtime |
| H1 | Research-informed hypothesis runtime |
| V1 | Behavior-verified runtime |
| P1 | Regression-checked pruned runtime and sole promotion candidate |

Historical B0/C1 campaign records retain their original meaning.

Prompt 1 freezes M0 from local intent, callers, relationships, compatibility,
authority, and safety before upstream packages, outside research, or current
skill bodies are opened. Prompt 3 materializes exact M0 bytes only from that
checkpoint. M0 is minimal by behavioral scope and must pass its complete
viability suite.

The Research Pass first searches independently from the intended behavior. It
then inspects upstream and current behavior and verifies newly observed
mechanics against applicable online professional evidence, alternatives,
conditions, and counterpressure. Upstream behavior proves what a pack
instructs, not professional correctness or a professional claim its author did
not make. Method evidence is classified `independently-supported`,
`contested`, `pack-specific`, or `unverified`.

H1 transforms M0 through admitted professional methods, current behavior,
upstream observations, pack composition, or intent-adjacent opportunities.
Pack-specific or unverified behavior may proceed only as an explicitly local
experiment when research found no decisive contradiction. Every H1 unit names
an owner, expected M0 weakness, contribution mode, cheapest expression,
wrong-condition case, and proof.

Prompt 4 proves M0 first. H1 may demonstrate defect correction or a
pre-registered quality lift. A missing control deficit rejects only that
hypothesis. An inconclusive material removal risk preserves current active
runtime. Accepted behavior becomes V1.

The Pruning Pass freezes V1 and derives P1 only through non-regression proof.
Unproved cuts revert to V1. Prompt 5 promotes and installs only P1.

A Conditional Behavior Decision Interlude uses `$grill-with-docs` only for a
user-owned contract choice. A Conditional Prototype Interlude resolves one
agent-owned construction choice. Research, intent confirmation, and Prototype
evidence may admit or shape a hypothesis; none proves exact steering effect.

`plan-only` work ends after Prompt 2. A new unscaffolded skill uses
`skill-creator` only for structure and metadata. Writing Great Skills owns
semantic quality; installation and Git delivery retain separate owners.

Bare campaign invocation excludes Git delivery. Append `and commit` or
`and push` to authorize Prompt 6.

## Note Types

Use whole-skill synthesis when source pressure and current-runtime evidence are
strong enough to select a coordinated design. Preserve the viability floor,
M0 checkpoint, research packet, current dispositions, H1 decisions, V1/P1
state, material alternatives, ownership, affected surfaces, and proof design.

Existing synthesis notes are reconciled in place only when their skill enters
the workflow. Do not mass-rewrite them or fabricate historical evidence.
Older source-map synthesis remains historical evidence unless an owning method
promotes it.

## Output

A whole-skill synthesis note should contain:

- orientation, viability floor, and hard ownership boundary;
- intent-derived M0 checkpoint, clause map, and executable definition;
- independent research registry, method classifications, alternatives,
  counterpressure, and professional claim limits;
- one ledger row per semantic behavior unit;
- current/M0/H1 identities and final V1/P1 state;
- current, upstream-observed, independently supported, contested,
  pack-specific, unverified, and locally experimental dispositions;
- H1 transformations with origin, contribution mode, expected M0 weakness,
  owner, cheapest expression, wrong condition, and proof;
- prior-evidence dispositions and exact reusable proof lanes;
- decision trace, rejected and deferred alternatives, affected-surface map,
  and deliberate non-changes;
- proof matrix separating M0 fidelity and viability, method evidence, H1
  defect correction or quality lift, protected behavior, deterministic proof,
  and V1/P1 pruning equivalence; and
- remaining research, behavioral, model, host, and transfer gaps.

## Rule

Research may be exhaustive. Synthesis is decision-complete, not
transcript-complete. Runtime is behavior-complete and compact. Validation owns
exact outputs and chronology. After promotion, reconcile synthesis to active
P1 and remove superseded construction detail without erasing
decision-changing provenance.
