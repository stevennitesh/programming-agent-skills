### Intended Contract

**Outcome:** transform one settled source packet into an exhaustive,
dependency-ordered set of bounded implementation tickets that satisfy the
configured tracker's Ready-for-agent contract and are safe for a fresh
implementation session to select.

**Invocation:** explicit-only. Admit a direct request to turn settled source
into dependency-ordered Ready-for-agent tickets. A verified parent spec, a
settled multi-slice improvement candidate, or a settled multi-item remediation
packet can supply the source.

**Exclusions:** do not synthesize or alter product intent, create a parent
specification, resolve user-owned decisions, triage raw intake, implement a
ticket, deliver a parent graph, dispatch workers, close work, install skills,
or perform Git delivery.

**Authority:**

- The supplied settled source and its Source Trace own outcome, commitments,
  acceptance, scope, and domain meaning.
- `to-tickets` owns implementation slicing, exhaustive coverage, dependency
  order, per-ticket readiness judgment, and the execution-profile judgment
  needed by its consumers.
- Tracker and label docs own transport, issue relationships, state/category
  mappings, lifecycle rules, and Mutation read-back.
- `implement` owns one selected ticket's delivery.
- `parallel-implement` owns an explicitly requested parent-delivery campaign
  after receiving a complete non-empty parent-backed graph.
- A missing or incompatible required setup surface belongs to
  `repo-bootstrap`.

**Caller and relationship obligations:**

- Preserve the complete source packet and sharp pointers to every relied-on
  source in each affected ticket's Source Trace.
- Preserve a verified parent as intent owner and link children through the
  configured parent/child convention.
- Produce standalone Ready-for-agent tickets when settled source authorizes
  slices but no tracker parent exists; do not fabricate a parent.
- Return an exact repair packet when an implementation consumer identifies
  incomplete or contradictory slicing, readiness, dependency, execution
  profile, or state-matrix information.
- Recommend downstream work and stop; never invoke an explicit-only delivery
  owner.

**Safe failure Return:**

- Missing/incompatible setup: name the exact surface, recommend
  `$repo-bootstrap`, preserve source and tracker, and stop before mutation.
- Unsettled source or a missing decision that would change a commitment:
  return the exact decision and affected slices without mutation.
- No justified implementation slice: return the coverage result and recommend
  no successor.
- Tracker mutation/read-back failure: report applied operations, failed
  operations, affected dependents, observed frontier, and the safest recovery
  action; claim no complete graph.
- Consumer repair return: preserve valid settled intent, identify every
  readiness/graph defect in one packet, and mutate only after the corrected
  graph is complete.

**Completion:** complete only when the source is settled; every commitment is
mapped to at least one ticket or an explicit out-of-scope/deferred
disposition; each ticket passes the Ready-for-agent contract; the dependency
graph, execution profiles, and applicable state matrices are complete and
non-contradictory; every authorized tracker mutation and affected relationship
reads back; and exactly one downstream recommendation or `none` is returned
without starting it.

**Irreversible order:** validate setup and source authority; freeze the entire
ticket and dependency graph in memory; verify coverage and readiness; only
then create tracker items; then apply parent/dependency relationships and
mapped roles; finally refetch every created/changed item and affected
dependent before claiming completion. Never publish partial design work as
ready.

**Compatibility and safety:**

- Use the repository-routed tracker, domain, engineering, and label contracts.
- Apply the mapped `ready-for-agent` state. Apply exactly one category role
  when the settled source authorizes it; do not invent category meaning.
- Preserve unresolved blockers as blockers; closure or partial mutation must
  never create a false-ready frontier.
- Treat expected write overlap, shared proof resources, trust boundaries,
  migrations, cutovers, permissions, protected data, and irreversible state
  as serial constraints unless the settled packet and proof make independence
  explicit.
- For each stateful ticket, include its applicable state-boundary matrix:
  absent/initial state, current reusable state, legacy/incompatible state,
  public access paths, supported variants, and relevant lifecycle transitions,
  reduced to the distinct supported branches and high-risk interactions.

   verification authority/evidence.

   and any node made falsely ready by an unresolved blocker.
8. Give every ticket an execution profile covering semantic ownership,
   expected production writes, proof seams and scarce proof resources,
   ordering, serial tripwires, and evidenced independence or serialization.
9. Give every stateful ticket its applicable state-boundary matrix; state
   `not applicable` with a reason for a stateless ticket.
10. Apply the configured `ready-for-agent` state only after the ticket passes
    every ready gate. Apply a category role only when settled source owns it.
11. Freeze and audit the whole graph before external mutation. Create tracker
    items, then relationships and roles, through tracker-owned operations.
12. Refetch every created/changed item and affected dependent. Verify exact
    body, relationships, roles, assignee, status, and resulting frontier.
    Report partial application and the safest recovery rather than claiming
    success on mismatch.
13. Return the verified ordered graph and exactly one recommendation:
    `$implement` for a single item or when overlap, a serial tripwire,
