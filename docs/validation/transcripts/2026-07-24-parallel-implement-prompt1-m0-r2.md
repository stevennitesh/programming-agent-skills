## Viability Floor

### Intended outcome

Finish the complete in-scope child graph of one settled parent, preserving the
parent and tickets as commitment authority, while the root orchestrator owns
delivery technique, concurrency qualification, dispatch, integration, formal
review, bounded Repair, campaign state, and Lock evidence.

### Invocation and exclusions

- Invocation is explicit-only. Admission requires an explicit top-level parent
  delivery request and a non-empty exhaustive Ready-for-agent graph produced
  by, or equivalent to the complete contract owned by, `to-tickets`.
- A single ready item or the first serial item is not this skill's scope;
  `implement` owns that delivery shape.
- Missing setup recommends `repo-bootstrap` and stops before tracker, Git, or
  production mutation.
- Incomplete or contradictory graph, readiness, execution-profile, or
  state-matrix input returns one exhaustive repair packet, recommends
  `to-tickets`, and stops without mutation.
- Parent-spec creation, source shaping, raw-intake triage, ticket slicing,
  tracker transport rules, domain truth, specialist implementation procedure,
  reviewer judgment, installation, push, and unrelated cleanup remain owned
  elsewhere.

### Authority

- The user and parent/ticket Source Trace own outcome, commitments, acceptance,
  supported paths, exclusions, category authorization, and commitment changes.
- `to-tickets` owns graph completeness, blocker order, ticket readiness,
  execution profiles, state matrices, and the initial truthful frontier.
- Tracker and label documents own external transport, claims, dependency state,
  lifecycle mutations, labels, closeout mechanics, and mutation read-back.
- The engineering contract owns Charter, commitment boundary, proof, work
  state, Repair generation, fixed-snapshot review language, and Lock.
- The root `parallel-implement` orchestrator alone owns campaign dispatch,
  concurrency/serialization judgment, integration order, formal-review
  selection, finding admission into Repair, and the terminal campaign Return.
- Lane workers and child integrators own only their bounded assigned mutation
  and evidence packets. They do not fan out, formally review, close tracker
  items, alter the Charter, integrate sibling work, push, or declare campaign
  completion.

### Caller and relationship obligations

- The caller supplies the settled parent identity, complete ordered child set,
  Source Trace, observable acceptance, dependency graph, proof lanes, expected
  write scopes, scope fences, readiness, execution profiles, applicable state
  matrices, and an explicit request to deliver the whole parent graph.
- Invoke `tdd` inside a lane for settled red-testable new behavior, or a bug
  whose expected behavior, exact symptom, cause, and trusted red-capable
  reproduction are known.
- Invoke `diagnosing-bugs` inside the same lane when any of those bug facts is
  uncertain; its causal packet returns to that lane.
- Invoke `review` for an ordinary drained proved candidate or repaired
  successor. Invoke `convergent-pr-review` instead for a local PR or bounded
  high-risk diff. Judgment returns to the root.
- Invoke `resolving-merge-conflicts` when serial landing enters preserved
  conflict or partial Git state. Resume only from its exact verified state and
  authorized next boundary.
- Review findings are admitted only through the shared finding contract. A
  complete blocking report remains intact for caller decision and grants
  neither mutation nor successor-snapshot authority. Repair opens only after
  the caller admits the complete blocking set and that entire set is
  automatic-in-scope, Charter-preserving, and within the frozen Repair Budget.

### Safe failure Return

Every stop preserves observed tracker and Git state and returns: the frozen
parent/graph identity, campaign identity, completed and uncompleted nodes,
claims and worktrees, accepted landing identities, current frontier,
dependency-overlay state, proof and review evidence, applied and failed
mutations, exact blocker, safest recovery action, residual risk, and whether a
resume is valid. Partial mutation is never reported as complete. A rejected or
invalidated landing removes its same-campaign execution overlay and reblocks
dependents.

### Completion

Complete only when every in-scope child has an accepted integrated commit,
required current proof, accepted formal review, verified tracker closeout, and
released claim; every Repair generation is within the frozen budget and itself
proved and reviewed; the final integrated state satisfies the parent Charter;
children close before the parent; the parent closeout and resulting frontier
read back; campaign state and Git state reconcile; disposable work is removed
or intentionally named; no unauthorized push or adjacent mutation occurred;
and the root returns evidence and residual risk.

### Irreversible order, compatibility, and safety

1. Verify setup and the complete caller packet before any mutation.
2. Freeze the Charter, fixed point, graph, identities, Repair Budget, mutation
   boundary, and campaign state before claims or dispatch.
3. Claim a child before dispatch; refresh and revalidate state after every
   external interaction or worker return.
4. Dispatch only qualified independent frontier work. Serialize overlap,
   scarce proof resources, trust boundaries, migrations, cutovers,
   permissions, protected data, irreversible state, and uncertain
   independence.
5. Workers change only isolated bounded lanes and return one commit plus proof.
   The root drains results, then lands them serially in dependency-safe order.
6. Validate each landing and the accumulated integration state. Preserve
   conflict/partial Git state and use the conflict owner rather than guessing.
7. Drain all implementation work before root-owned formal review. Return a
   mixed blocking report intact for caller decision. Open Repair only after the
   caller admits the complete blocking set and the entire set is
   automatic-in-scope, Charter-preserving, and within budget; review each
   authorized repaired successor.
8. Close verified children first and the parent last, with tracker mutation
   read-back after each affected transition.

Compatibility includes the configured tracker contract; Git working tree,
index, committed tree, isolated worktrees/branches, and partial conflict state;
the repo's canonical checks; parent/child and blocker representations; the
campaign-scoped `landed-awaiting-lock` overlay; and target-owned worker,
integrator, launch, ledger, and finding interfaces. The neutral M0 does not
select undocumented helper syntax or a new storage format.
