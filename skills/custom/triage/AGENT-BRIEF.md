# Writing Codex-Ready Briefs

A Codex-ready brief renders the tracker's Ready-for-agent contract when an issue or PR moves to `ready-for-agent`. The original request and discussion remain source context; the brief names one bounded handoff.

For an issue, the brief describes the slice to build. For a PR, it describes the slice left to finish on the existing diff.

## Shape

Prefer one independently completable tracer bullet through real behavior. Use a support slice only when it independently proves, unblocks, or de-risks later behavior. Describe durable behavior and interfaces rather than current file choreography; make acceptance observable through the highest useful seam and fence adjacent work.

## Template

```markdown
> *This was generated by AI during triage.*

## Codex-Ready Brief

**Category:** bug / enhancement
**State:** ready-for-agent
**Slice type:** tracer bullet / support slice
**Summary:** one-line description of the bounded slice

**Current behavior:**
Describe the verified status quo. For a PR, describe the current state of the diff.

**Verification evidence:**
- Result: confirmed / partial / maintainer override
- Evidence: reproduction, command result, code path, or PR diff evidence

**Desired behavior:**
Describe the observable behavior after this slice, including edge cases, error conditions, and user-visible or operator-visible outcomes.

**Why this slice:**
For a tracer bullet, name the behavior or risk it proves.
For a support slice, name what it unblocks or de-risks.

**Relevant interfaces or seams:**
- Interface, command, workflow, adapter, config shape, or contract - why it matters

**Relevant context:**
- Domain terms: term or `none`
- ADRs: path or `none`

**Source Trace:**
- Request and decision-bearing tracker comments
- Prior triage notes
- Relevant code or PR diff
- Domain glossary and ADR pointers
- Prior rejection record or `none`

**Dependencies and parallel safety:**
- Parent: issue, spec, packet, or `none`
- Blocked by: issue, PR, decision, or `none`
- Expected write scope: durable modules, interfaces, commands, or docs
- Parallel safety: independent after blocker lands / likely sibling overlap / serialize because...

**Acceptance criteria:**
- [ ] Specific, observable criterion 1
- [ ] Specific, observable criterion 2
- [ ] Specific, observable criterion 3

**Proof lane:**
Name the highest useful interface or seam to prove through.
For support slices, name the observable proof: migration check, harness output, compatibility proof, config behavior, or repo-standard command.
When load-bearing internals determine correctness, name the semantic examples, fixtures, invariants, or edge cases.

**Out of scope:**
- Adjacent behavior that stays unchanged
- Related feature, cleanup, or follow-up that belongs elsewhere
```

## Ready Gate

Complete only when the brief names one bounded slice; verification is confirmed or a maintainer override and its residual uncertainty are explicit; Source Trace is preserved; dependencies, write scope, and parallel safety are explicit; acceptance criteria are observable; the proof lane reaches the highest useful seam; and adjacent work is fenced out.
