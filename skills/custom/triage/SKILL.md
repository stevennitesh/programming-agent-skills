# Triage

Triage is the **intake state machine** for raw issues and configured external PR requests.

Tracker docs own transport and label mechanics. Triage owns classification, request verification, shaping, state mutation, ready briefs, and rejected-enhancement memory. `$implement` and `$parallel-implement` own code changes and `implemented` closeout.

Apply PR triage only when `docs/agents/issue-tracker.md` enables PRs as a request surface. A named PR bypasses only the external-author discovery filter. For triage semantics, a **PR is an issue with attached code**.

**Disclaimer gate:** prefix every issue or comment posted to the tracker during triage with:

```markdown
> *This was generated by AI during triage.*
```

## Contracts

Read `docs/agents/issue-tracker.md` and `docs/agents/triage-labels.md` before tracker access. A missing contract blocks triage; recommend `$setup-matt-pocock-skills`.

When codebase context matters, follow `docs/agents/domain.md` to the relevant glossary and ADRs. Before reproduction, checkout, or executable validation, read `docs/agents/engineering-contract.md` when present.

Read [AGENT-BRIEF.md](AGENT-BRIEF.md) only when preparing a `ready-for-agent` or `ready-for-human` brief. Read [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) only when screening, rejecting, or reconsidering an enhancement.

Treat `$to-tickets` output as already ready-for-agent. Re-enter triage only when the maintainer explicitly asks to correct its state or brief.

## State Machine

**Role invariant:** every completed item carries exactly one mapped category role and one mapped state role. Surface missing or conflicting roles before mutation.

Normal intake moves from unclassified to `needs-triage`, then to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. Reporter activity returns `needs-info` to `needs-triage`.

For a PR, `ready-for-agent` means a Codex-ready brief names the next bounded slice on the diff. `ready-for-human` means the PR is ready for human judgment or merge.

`implemented` is a closeout state owned by `$implement` or `$parallel-implement`. Triage may apply it only as an explicit label correction backed by existing closeout evidence.

**Ready gate:** `ready-for-agent` requires a bounded slice, Source Trace, verification evidence, acceptance criteria, proof lane, dependency state, write scope, parallel-safety note, and scope fence. Record an explicit label-only override in the triage packet.

## Choose One Branch

- **Attention scan:** show incoming work that needs maintainer attention.
- **Specific item:** verify, shape, recommend, and apply one outcome.
- **Quick override:** apply an explicitly named state outcome with reduced discovery.

## Attention Scan

Query the configured tracker and present these buckets oldest first:

1. **Role drift:** unlabeled items and items with missing or conflicting category or state roles.
2. **`needs-triage`:** evaluation in progress.
3. **`needs-info` with reporter activity:** reporter activity since the latest triage note.

Assign each item to its first matching bucket so counts stay disjoint.

When PR triage is enabled, include configured external PRs and tag every line `[PR]` or `[issue]`.

Show counts and one-line summaries, make no mutations, and let the maintainer choose.

Attention scan is complete when all three buckets were evaluated, PR policy was respected, counts and summaries were returned, and tracker state stayed unchanged.

## Specific Item

### 1. Trace

Build the **Source Trace** from:

- the current request;
- tracker and label contracts;
- the full issue or PR body, labels, author, dates, decision-bearing comments, and prior triage notes;
- the PR diff when applicable;
- relevant glossary terms and ADRs;
- relevant code and verification evidence;
- matching rejected-enhancement records.

Parse prior triage notes so resolved questions remain resolved.

Run two screens:

- **Redundancy:** search by domain concept for existing behavior and report where you looked.
- **Prior rejection:** screen `.out-of-scope/` for concept matches and load likely matches in full.

Tracing is complete when the claim, prior decisions, current roles, relevant implementation surface, and evidence gaps are known.

### 2. Verify

- **Bug:** reproduce the reported behavior and return `confirmed`, `failed`, or `insufficient detail`, with the failing behavior and likely code path.
- **Enhancement:** verify current behavior, relevant seams, plausibility, redundancy, and prior rejection.
- **PR:** inspect the tracker-provided diff first. When execution is required, use an isolated worktree or approved clean checkout and run focused commands.

Verification ends with evidence. Route code changes to implementation.

### 3. Shape

Reporter-owned facts that are missing route directly to `needs-info`.

When maintainer-owned scope, acceptance, domain language, or design decisions remain unresolved, run `$grill-with-docs`. Refresh the Source Trace and affected verification after shaping.

### 4. Recommend

Return the proposed category and state with:

- reasoning;
- current behavior and relevant seams;
- redundancy and prior-rejection results;
- verification result;
- remaining unknowns;
- proposed tracker, comment, brief, `.out-of-scope/`, and close mutations.

Wait for maintainer direction.

### 5. Mutation Gate

Maintainer direction approves the recommended outcome.

Before acting, state one **mutation packet**:

- roles before and after;
- mapped labels added and removed;
- comment or brief;
- `.out-of-scope/` create, update, or delete;
- open or closed state.

Ask again only when the packet adds an unapproved mutation or the recommended outcome changed after verification or shaping.

### 6. Apply

Replace prior role labels so the role invariant holds.

- `ready-for-agent`: post the brief from [AGENT-BRIEF.md](AGENT-BRIEF.md), unless the maintainer explicitly requested label-only.
- `ready-for-human`: use the same brief fields under `## Human-Ready Brief` and state why human judgment, access, design, or testing is required.
- `needs-info`: post the Needs-Info template.
- `wontfix`, already implemented: point to the existing behavior, then close.
- `wontfix`, rejected bug: explain the rejection, then close.
- `wontfix`, rejected enhancement: update `.out-of-scope/`, link it from the comment, then close.
- `needs-triage`: apply the state and preserve meaningful partial progress in a comment.

Apply actual label strings through `docs/agents/triage-labels.md`.

### 7. Read Back

Fetch the item after mutation. Verify:

- exactly one category and one state;
- expected comment or brief exists with the disclaimer;
- close state matches the mutation packet;
- required `.out-of-scope/` changes exist.

A partial mutation is blocked, not complete. Report the applied operations, failed operations, and safest recovery action.

Return a **triage packet** containing Source Trace, verification, roles before and after, comment or brief reference, changed local paths, close state, skipped checks, blockers, and next route.

## Quick Override

Use this branch when the maintainer names the exact target state.

Read current roles and enough source context to produce any required comment or brief. Skip request verification and grilling.

The command approves the named outcome and its stated mutation packet. Keep the normal ready brief, disclaimer, `.out-of-scope/`, close, and read-back gates.

For `implemented`, require an explicit label-correction request and existing closeout evidence.

## Completion Criteria

Attention scan completes through its read-only branch criterion.

A mutation branch completes only when the maintainer-approved packet was applied, read back, and returned as a triage packet; role invariants hold; every tracker comment carries the disclaimer; required briefs and rejected-enhancement memory exist; and partial failures, skipped checks, and blockers are reported.
