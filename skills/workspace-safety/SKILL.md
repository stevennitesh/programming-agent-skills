# Workspace Safety

Protect user and agent source changes. Make repo operations reversible whenever practical.

## Rule

Existing source, tests, docs, config, workflows, fixtures, migrations, lockfiles, generated files, and untracked paths belong to the user unless you created the change in this session. Preserve unknown repo work before optimizing speed.

When this skill triggers, first name the operation and path types at risk: source, tests, docs, config, workflow, fixture, migration, lockfile, generated output, scratch, branch/worktree, staging, commit, PR, dependency install, or destructive action. This changes the route from "run the command" to "prove the command will not overwrite or mix unrelated work."

Required baseline before risky work:

- Current branch and dirty state from `git status --short --branch` when Git is available.
- Relevant dirty or untracked paths classified as mine, user-provided, generated by a command, scratch output, or unknown.
- Relevant hunks inspected before touching a dirty tracked file.
- Scratch, cache, generated-output, or worktree paths checked as ignored or kept outside the repo before relying on them.
- Stop condition identified when unknown changes overlap the same lines, public or caller contract, state/data path, generated output, dependency/config behavior, or branch/worktree operation.

Do not assume a dirty or untracked file is yours because it is convenient, expected, or probably generated.

## Baseline

Before risky source edits, generated output, dependency changes, branch/worktree actions, staging, commits, or other git actions when git is available:

```bash
git status --short --branch
```

If files are dirty:

- Classify relevant path changes as yours, user-provided, generated by a command, scratch output, or unknown.
- Do not revert or overwrite unknown changes.
- If touching a dirty source, test, docs, config, workflow, fixture, migration, or lockfile path, inspect the relevant hunks first and work with them.
- If your edit and an unknown edit affect the same lines, public or caller contract, state/data path, generated output, or dependency/config behavior, stop and ask or choose a smaller edit.
- Treat untracked source, tests, docs, config, fixtures, migrations, lockfiles, workflows, scripts, and generated files as possible user work unless they are known scratch output you created.
- Record branch and head SHA when commit, PR, CI, or worktree state will matter later.

If Git is unavailable or the command cannot show status, say what workspace evidence is missing and use the smallest safer file-level read before editing.

## Scratch

- Use `.tmp/` for scratch files, generated issue/PR bodies, command logs, reproduction notes, caches, and local experiment output.
- Do not commit scratch output unless requested.
- `CONTEXT.md` is durable repo vocabulary, not scratch. Inspect it before editing if dirty.
- Before relying on a new repo-local scratch, generated-output, cache, or worktree directory, verify it is ignored with repo evidence such as `git check-ignore -v <path>` or `git status --ignored --short <path>`, or keep it outside the repo.
- If a scratch path is not ignored, move it to an ignored path or stop before creating local-only output in tracked scope.

## Branches And Worktrees

Use a branch or worktree when:

- Work touches many source modules, migrations, generated outputs, or unrelated areas.
- Multiple implementation, refactor, bugfix, or verification slices may run independently.
- User wants isolated feature, refactor, bugfix, or PR work.
- Current tree has unrelated source, test, config, docs, workflow, or generated-file changes.

For repo-local worktrees:

- Ensure the directory is ignored before creating it.
- Check the current branch and worktree list before removing or switching.
- Run the smallest known baseline check in a new worktree when the work depends on distinguishing new failures from pre-existing test, build, lint, or runtime failures.
- Do not remove a worktree unless it was created for this task and no needed changes remain, or the user approves.
- For parallel implementation, use `worktree-isolation` before editing and keep ownership scope explicit: files/modules, public or caller contracts, generated output, dependency/config state, and integration owner.

## Git Safety

Never run destructive commands unless explicitly requested and the affected source, generated, scratch, and git paths are understood:

- `git reset --hard`
- `git clean`
- `git checkout -- <path>`
- `git restore <path>`
- branch deletion
- force push
- recursive delete or move of files

If `git clean` is approved, dry-run first with `git clean -n` and compare the listed paths to the user's intent. Avoid `-x` unless ignored files are explicitly part of the cleanup.

Prefer non-interactive git commands. Show exact risk, target paths, source scope, and whether the action can be undone when asking for approval.

Do not soften destructive commands with assumptions. If the request is ambiguous, inspect first, ask with the target paths and risk, and avoid running the command until the affected scope is clear.

## Generated Or Installed Output

Dependency installs, formatters, code generators, migrations, and build tools can change source, fixtures, schemas, lockfiles, package metadata, workflow files, or generated output. Before keeping those changes:

1. Know which requested behavior, repo check, or reproducibility need requires the command.
2. Inspect the new diff by path type: source, tests, docs, config, lockfile, generated output, migration, or workflow.
3. Keep only outputs that support the requested work, acceptance check, or repeatable verification.
4. Do not delete unknown generated files, lockfile changes, caches, or local artifacts unless you created them or the user approves.
5. If generated output or dependency changes appear outside the expected path types, stop and classify them before continuing.

## Commit/PR Readiness

Before commit or PR:

1. Review the working-tree diff by changed path and source scope.
2. Run relevant checks after the final source, test, config, docs, workflow, or generated-file change.
3. Stage only intended source, test, docs, config, workflow, lockfile, migration, or generated files; avoid broad staging in a dirty tree.
4. Use a message that states the real behavior, source, or repo-workflow change.
5. Leave unrelated user or agent work untouched.
6. Use `verify-before-done` before claiming the branch, commit, or PR is ready.

## Stop Or Ask

Stop or ask before editing, staging, committing, switching, deleting, installing, or generating output when:

- `git status --short --branch` shows dirty or untracked paths that may overlap the requested work and you have not classified them.
- A dirty tracked file must be touched but its relevant hunks have not been inspected.
- Unknown changes overlap the same lines, public or caller contract, state/data path, generated output, dependency/config behavior, migration, workflow, or branch/worktree operation.
- Untracked source, tests, docs, config, workflows, fixtures, migrations, lockfiles, scripts, or generated files could be user work.
- Scratch, cache, generated output, or worktree paths are inside the repo and not proven ignored.
- A formatter, dependency install, migration, generator, cleanup command, or broad test/build command produced unexpected source, config, lockfile, workflow, fixture, migration, or generated-output changes.
- A destructive Git or filesystem operation is requested without clear target paths, source scope, generated/scratch impact, and approval.
- Branch, commit, PR, CI, merge, or worktree state matters but current branch, head SHA, remote/PR state, or worktree list is unknown.
- The safest path is a smaller edit, separate worktree, local self-review, or user decision.

## Handoff

- Return to the controlling skill after the safety risk is handled.
- Use `worktree-isolation` when branch/worktree safety is needed because approved implementation will run in parallel across agents or sessions.
- Use `github-tracking` for issues, PRDs, PR creation, review threads, or CI/check evidence.
- Use `verify-before-done` before any final ready, fixed, passing, safe, or mergeable claim.
