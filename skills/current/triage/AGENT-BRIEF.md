# Writing Codex-Ready Briefs

A Codex-ready brief is a structured comment posted on an issue or PR when it moves to `ready-for-agent`. It is the contract an unattended Codex implementation session or delegated implementation subagent works from. The original body and discussion are context; the brief is the handoff.

For an issue, the brief describes the bounded slice to build. For a PR, it describes the bounded slice left to finish on the existing diff.

## Principles

### One bounded slice

A ready-for-agent brief should describe one bounded slice.

Prefer a **tracer bullet** for product behavior: one narrow behavior through the real system with observable acceptance criteria.

Use a **support slice** only when it clearly unblocks or de-risks implementation and can be independently validated, such as behavior-preserving prefactoring, migration work, test harness work, compatibility proof, config, or operational wiring.

### Durable over procedural

The brief may sit in `ready-for-agent` for days or weeks. Write it so it stays useful even as files are renamed, moved, or refactored.

Describe behavior, interfaces, seams, contracts, types, commands, config shapes, and validation. Avoid line numbers and step-by-step implementation instructions.

### Complete acceptance criteria

Codex needs to know when it is done. Every brief must have concrete, observable acceptance criteria. Each criterion should be independently verifiable.

### Explicit scope boundaries

State what is out of scope. This prevents gold-plating and keeps the bounded slice intact.

## Template

```markdown
> *This was generated by AI during triage.*

## Codex-Ready Brief

**Category:** bug / enhancement
**State:** ready-for-agent
**Slice type:** tracer bullet / support slice
**Summary:** one-line description of the bounded slice

**Current behavior:**
Describe what happens now. For bugs, this is the broken behavior.
For enhancements, this is the status quo the slice builds on.
For PRs, describe the current state of the diff.

**Desired behavior:**
Describe what should happen after the bounded slice is complete.
Be specific about edge cases, error conditions, and user-visible or operator-visible outcomes.

**Why this slice:**
Explain why this is the right bounded slice now.
For tracer bullets, name the behavior or risk it proves.
For support slices, name what it unblocks or de-risks.

**Relevant interfaces or seams:**
- `InterfaceName` or seam - why it matters
- Config shape, command, workflow, adapter, or contract - expected behavior

**Acceptance criteria:**
- [ ] Specific, observable criterion 1
- [ ] Specific, observable criterion 2
- [ ] Specific, observable criterion 3

**Validation notes:**
Describe how Codex should prove the slice is complete.
For tracer bullets, name the highest useful interface or seam to test through.
For support slices, name the observable validation: migration check, harness output, compatibility proof, config behavior, or repo-standard command.
If load-bearing internal behavior determines correctness, name the semantic examples, fixtures, invariants, or edge cases Codex should use to prove it.

**Out of scope:**
- Adjacent behavior that should not be changed
- Related feature, cleanup, or follow-up that should be separate
```

## Examples

### Good Codex-ready brief: bug tracer bullet

```markdown
> *This was generated by AI during triage.*

## Codex-Ready Brief

**Category:** bug
**State:** ready-for-agent
**Slice type:** tracer bullet
**Summary:** Skill description truncation should not cut words in half

**Current behavior:**
When a skill description exceeds the description length limit, it is truncated at exactly the limit regardless of word boundaries. This can produce broken descriptions that end mid-word, such as `Use when the user wants to confi`.

**Desired behavior:**
Long descriptions should truncate at the last word boundary that fits within the limit and append `...`. Descriptions under the limit should stay unchanged.

**Why this slice:**
This proves the user-visible description formatting behavior end to end without changing the length limit or description parsing model.

**Relevant interfaces or seams:**
- Skill metadata loading - produces the description shown to Codex and users.
- Description truncation behavior - should preserve whole words and the existing max length contract.

**Acceptance criteria:**
- [ ] Descriptions under the limit are unchanged.
- [ ] Descriptions over the limit truncate at the last word boundary that fits.
- [ ] Truncated descriptions end with `...`.
- [ ] The final description, including `...`, does not exceed the existing limit.

**Validation notes:**
Add or update a focused test through the skill metadata loading interface. The test should prove both unchanged short descriptions and word-boundary truncation for long descriptions.

**Out of scope:**
- Changing the length limit.
- Supporting multiline descriptions.
- Rewriting frontmatter parsing.
```

### Good Codex-ready brief: enhancement tracer bullet

```markdown
> *This was generated by AI during triage.*

## Codex-Ready Brief

**Category:** enhancement
**State:** ready-for-agent
**Slice type:** tracer bullet
**Summary:** Triage list should show issues that need maintainer attention

**Current behavior:**
Maintainers have to manually inspect issue labels and comments to find work that needs triage. There is no single command or view that groups unlabeled issues, active `needs-triage` issues, and `needs-info` issues with reporter replies.

**Desired behavior:**
When a maintainer runs the triage list flow, they should see three oldest-first buckets: unlabeled issues, `needs-triage` issues, and `needs-info` issues with reporter activity since the last triage note. Each item should show its issue number, title, labels, and one-line summary.

**Why this slice:**
This proves the core maintainer workflow end to end: finding the next issue that needs attention.

**Relevant interfaces or seams:**
- Issue tracker query interface - fetches issues, labels, comments, and updated timestamps.
- Triage list presentation - groups and summarizes issues for the maintainer.

**Acceptance criteria:**
- [ ] Unlabeled issues appear in an `Unlabeled` bucket, oldest first.
- [ ] Issues labeled `needs-triage` appear in a `needs-triage` bucket, oldest first.
- [ ] Issues labeled `needs-info` appear only when the reporter has replied since the last triage note.
- [ ] Each listed issue includes number, title, labels, and a one-line summary.

**Validation notes:**
Test through the triage list interface with fixture issues covering all three buckets, including a `needs-info` issue with and without reporter activity.

**Out of scope:**
- Changing issue labels.
- Posting comments.
- Triage of a selected issue.
- PR discovery.
```

### Good Codex-ready brief: support slice

```markdown
> *This was generated by AI during triage.*

## Codex-Ready Brief

**Category:** enhancement
**State:** ready-for-agent
**Slice type:** support slice
**Summary:** Add a local issue-tracker fixture harness for triage tests

**Current behavior:**
Triage behavior depends on issue tracker state, but tests have to hand-roll tracker responses inside each test. This makes it hard to prove triage buckets, reporter activity, and label transitions consistently.

**Desired behavior:**
There should be a reusable local fixture harness for triage tests that can represent issues, labels, comments, authors, timestamps, and PR/issue distinction without calling a live tracker.

**Why this slice:**
This support slice de-risks later triage tracer bullets by giving them a stable local substitute for issue tracker behavior.

**Relevant interfaces or seams:**
- Issue tracker adapter seam - the harness should satisfy the same behavior tests need from the real tracker.
- Triage test fixtures - should make bucket and state-transition tests concise.

**Acceptance criteria:**
- [ ] Tests can create issue and PR fixtures with labels, comments, authors, and timestamps.
- [ ] Tests can represent reporter activity after the last triage note.
- [ ] The harness avoids live network calls.
- [ ] At least one existing or new triage test uses the harness.

**Validation notes:**
Run the focused triage test file. The proof is a passing test that uses the harness to exercise issue tracker behavior without network access.

**Out of scope:**
- Replacing the production issue tracker adapter.
- Adding new triage behavior.
- Supporting every field exposed by the live tracker.
```

### Good Codex-ready brief: PR finish slice

```markdown
> *This was generated by AI during triage.*

## Codex-Ready Brief

**Category:** enhancement
**State:** ready-for-agent
**Slice type:** tracer bullet
**Summary:** Finish the contributor's `--json` output flag for `triage list`

**Current behavior:**
The PR adds a `--json` flag that serializes the triage list output to JSON. The happy path is present, but two gaps remain: errors are still printed as human text, and the new flag has no test coverage.

**Desired behavior:**
With `--json`, triage list should emit well-formed JSON on stdout for both success and error cases. Exit codes should match the non-JSON command. Without `--json`, existing human-readable output should be unchanged.

**Why this slice:**
This finishes one bounded behavior already started by the PR: JSON output for the triage list command.

**Relevant interfaces or seams:**
- Triage list command interface - should preserve existing non-JSON behavior.
- JSON output path - should serialize success and error responses consistently.

**Acceptance criteria:**
- [ ] `triage list --json` emits valid JSON for success output.
- [ ] `triage list --json` emits valid JSON for error output.
- [ ] Exit codes match the non-JSON command.
- [ ] Default non-JSON output is unchanged.
- [ ] Tests cover one success case and one error case.

**Validation notes:**
Run the focused command tests for triage list. Include assertions for JSON parseability and unchanged non-JSON output.

**Out of scope:**
- Adding `--json` to other commands.
- Changing the JSON shape of the success payload already introduced by the PR.
- Reworking unrelated command output.
```

### Bad Codex-ready brief

```markdown
## Codex-Ready Brief

**Summary:** Fix the triage bug

**What to do:**
The triage thing is broken. Look at the main file and fix it.
The function around line 150 has the issue.

**Files to change:**
- src/triage/handler.ts line 150
- src/types.ts line 42
```

This is bad because it has no category, no state, no slice type, no current or desired behavior, no acceptance criteria, no validation notes, no scope boundaries, and it relies on stale file paths and line numbers.
